About the project

An end-to-end credit risk ML system

CreditIQ was built as a reference implementation of a production-shaped machine learning application: data generation, cleaning, feature engineering, model comparison, and a deployable web layer — all in one repository.

Request lifecycle

1. Applicant data submitted

Via the web form (POST /predict) or the JSON API (POST /api/predict).

2. Validation & cleaning

Required fields, numeric ranges, and type coercion are checked server-side before anything touches the model.

3. Feature engineering

Debt-to-income, loan-to-income, savings ratio, credit utilization risk, and employment stability are computed from the raw fields.

4. Encoding & scaling

Categorical fields pass through the label encoders fit during training; all numeric features are standardized with the saved StandardScaler.

5. Inference

The saved best model (selected by test-set ROC–AUC) scores the vector and returns a class + probability.

6. Explained response

Probability, confidence, a risk band, and a plain-language recommendation are assembled and returned.

Tech stack

Python 3.11 scikit-learn pandas / NumPy XGBoost LightGBM CatBoost imbalanced-learn (SMOTE) Flask Chart.js Bootstrap 5 Docker Render Joblib

Honest limitations

  • The dataset is synthetic, generated to mirror the schema and relationships of real-world credit approval datasets (income, credit score, DTI, employment stability, default history) — it is not real applicant data.
  • This is a decision-support demo, not a compliant lending system. Real credit decisioning requires fair-lending review, regulatory compliance (e.g. ECOA, FCRA in the US), and human oversight.
  • Authentication, a persistent database, PDF export, and a conversational assistant were intentionally left out of this pass to keep the core ML pipeline and UI solid — see the README's "Future Improvements" section.
